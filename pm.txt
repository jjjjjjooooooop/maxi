«Разработка программных модулей» 
 
 
. 
 
 
В данной инструкции ты найдешь информацию о том, как подготовиться к итоговой 
аттестации по дисциплине «Разработка программных модулей» в 5 семестре. 
Какие темы нужно повторить? 
Реализация итогового проекта подразумевает под собой создание веб -приложения «Чат -бот 
поддержки», где пользователь пишет в чат и получает ответ бот а. Тематика бота выбирается 
самостоятельно. При создании должны быть учтены следующие требования:   
 
API и корректность эндпоинтов   
1. Реализованы минимально необходимые эндпоинты, например:  
• POST /auth/register – регистрация нового пользователя  
• POST /auth/login – авторизация пользователя  
• POST /chat/message – принять сообщение пользователя, вернуть ответ бота, сохранить 
оба сообщения  
• GET /chat/history/{session_id} – выдать историю диалога  
• POST /chat/session – создать новую сессию  
2. Эндпойнты реализованы с использованием асинхронных функций и аннотаций типов  
3. Допускается реализация диалога через WebSocket-подключение (вместо эндпойнта POST 
/chat/message), при этом:  
• Реализован маршрут WebSocket /ws/chat (или аналогичный), обеспечивающий 
двустороннюю связь между клиентом и сервером  
• Обмен сообщениями происходит в формате JSON  
• Каждая пара сообщений (пользовательское и ответное) сохраняется в базе данных  
• При подключении по WebSocket осуществляется проверка JWT-токена и связывание 
соединения с пользователем и его сессией  
  
Модель данных и миграции   
1. Реализованы минимально необходимые SQLAlchemy-модели, например:  
• Session – сущность диалога, которая хранит идентификатор сессии и дату создания, а 
также связывает ее со всеми сообщениями и, при наличии, пользователем  
• Message – отдельное сообщение в рамках сессии с указанием отправителя, текстом и 
временем отправки  
• User – учетная запись пользователя с данными для аутентификации и привязкой к его 
сессиям  
   
2. Реализованы Alembic-миграции, база данных инициализируется с помощью миграций  
  
Валидация и обработка ошибок   
1. Реализованы Pydantic-схемы запросов и ответов, обеспечивающие валидацию обязательных 
полей, длины строк и отсутствие пустых значений  
2. Обеспечена корректная обработка исключений, таких как отсутствие сессии, передача 
пустого текста сообщения или некорректного ID сессии

Логика бота   
1. Реализована обработка пользовательских сообщений с использованием ключевых слов 
и/или шаблонов, предусмотрен ответ по умолчанию при отсутствии совпадений  
2. Осуществляется сохранение обоих видов сообщений – пользовательского и ответного – в 
историю диалога  
  
Аутентификация и сессии   
1. Реализованы механизмы регистрации и аутентификации пользователей, обеспечивающие 
безопасное хранение паролей с использованием криптографического хэширования, а также 
выдачу и проверку JWT-токенов для идентификации и авторизации при обращении к 
защищённым эндпойнтам   
2. Доступ к истории сообщений ограничен авторизованным пользователем, при этом 
предусмотрена проверка принадлежности сессии текущему пользователю  
  
Тестирование   
1. Реализованы 2–3 (минимум) автоматизированных теста, проверяющих создание сессии, 
отправку сообщения (включая сохранение обоих сообщений) и получение истории переписки   
2. Тесты выполняются в изолированной среде с использованием временной базы данных 
и фикстур  
  
Логирование и структура проекта   
1. Реализовано логирование запросов и ошибок   
2. Структура проекта организована в модульном виде, предусматривающем разделение на 
каталоги  
  
Документация и запуск   
1. В файле README представлена подробная инструкция по установке зависимостей, 
применению миграций и запуску приложения, а также приведены примеры основных запросов к 
API   
2. Описание и структура API задокументированы с использованием встроенных 
инструментов FastAPI (например, Swagger UI)  
 
Вы можете повторно обратиться к пройденным урокам в курсе в вашем ЛК, чтобы тщательно 
подготовится к созданию проекта.  
Формат аттестационной работы 
Итоговая аттестация по дисциплине «Разработка программных модулей» проходит в формате 
защиты проекта. Проект защищается перед однокурсниками и преподавателем, во время 
защиты будут задаваться вопросы по коду проекта. Защита является обязательной.  
Критерии оценивания работы 
За каждый блок выставляется по 10 баллов, суммарно – 80 баллов за весь проект. Каждый 
отдельный пункт в блоке оценивается по 5 баллов.
